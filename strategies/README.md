# 2.  Архитектура стратегии

Система построена по принципу последовательной фильтрации. Вход в позицию происходит только при одновременном выполнении всех активных фильтров. Каждый уровень независим и может быть включён или выключён отдельно.


## 2.1  Цепочка принятия решения

MACRO E1  →  IPDA ZONE  →  WEEKLY SWEEP  →  MACRO E2  →  ADX E3  →  RVOL E4  →  BTC E5
DXY/SPX      Discount/      Ликвидация      VIX/MOVE/    SLEEP/       Объём       BTC тренд
направление  Premium        недельного      JPY/EMA      TREND/       выше        и ETH/BTC
зона           уровня          фильтр       OVER         нормы       ratio

Если хотя бы один активный фильтр не выполнен — сигнал не генерируется, событие записывается в диагностический лог как BLOCKED с указанием причины.


## 2.2  Описание блоков

### Блок E1 — Macro Triad (DXY, SPX, EMA Confluence)
Базовый макроэкономический фильтр. Определяет общее направление рынка.

- DXY (Индекс доллара США): если доллар слабее своей скользящей средней — условие для LONG выполнено
- SPX (S&P 500): если акции выше своей средней — рисковый аппетит есть
- EMA Confluence (опционально): требует выравнивания EMA20 < EMA50 для DXY и EMA20 > EMA50 для SPX
- macro_ok = dxy_bear AND spx_bull — главное булево условие системы

### Блок — IPDA Zones (Discount / Premium)
Определяет позицию цены относительно диапазона последних N баров.

- Lookback 20/40/60 баров — глубина исследуемого диапазона
- EQ (Equilibrium) = (High + Low) / 2 — середина диапазона
- Discount: цена ниже EQ → зона покупок для LONG сигналов
- Premium: цена выше EQ → зона продаж для SHORT сигналов
- Позиция pos_pct (0–100%) записывается в каждую строку лога

### Блок — Weekly Sweep Detection
Детектор ликвидации недельных уровней. Подтверждает что цена сметала ликвидность прошлой недели.

- Берёт High и Low предыдущей полной торговой недели
- Если цена коснулась любого из уровней — weekly_sweep_hit = true
- Сбрасывается в false каждую новую неделю автоматически
- Обязательное условие: без sweep входа нет ни для LONG ни для SHORT

### Блок E2 — Macro Extended (VIX, MOVE, JPY)
Расширенный макрофильтр. Блокирует входы в периоды рыночной паники и нестабильности.

- VIX: при VIX > 20 статус WARN, при VIX > 35 статус PANIC (жёсткая блокировка)
- MOVE (ICE Bond Volatility): при MOVE > 140 рынок облигаций нестабилен
- JPY: резкое укрепление иены за 5 дней сигнализирует о бегстве в защитные активы
- Статистически подтверждён: при VIX WARN win rate = 38% (13 LOSS / 8 WIN за 2023-2026)

### Блок E3 — ADX Engine (Sleep / Trend / Overheat)
Фазовый анализ тренда на основе реального индикатора DMI (ta.dmi в Pine v6).

- SLEEP (ADX < 20): консолидация, тренда нет — опциональная блокировка входов
- TREND (ADX 20–40): активный направленный тренд — оптимальная фаза для входа
- OVER (ADX > 40): перегрев, тренд исчерпывается — опциональная блокировка
- ADX фаза также влияет на ширину Stop Loss в режиме SL/TP = ADX

### Блок E4 — RVOL (Relative Volume)
Фильтр относительного объёма. Подтверждение движения объёмом.

- RVOL = текущий объём / средний объём за N баров
- Результат исследования: на 4H RVOL неэффективен — использовать MOVE вместо
- На 1D рекомендованный порог: 1.2–1.5

### Блок E5 — BTC Correlation
Криптоспецифичный фильтр. Учитывает состояние BTC как ведущего актива крипторынка.

- BTC Trend: BTC price > BTC SMA(24) — бычий тренд BTC подтверждён
- ETH/BTC Ratio: отношение ETH к BTC выше своей средней — ETH сильнее BTC
- Находится на стадии сбора статистики. Оба переключателя OFF до накопления данных

# 3.  Система управления рисками (SL/TP Engine)

Стратегия поддерживает три независимых режима расчёта Stop Loss и Take Profit. Режим выбирается через параметр SL/TP Mode.


## 3.1  Режим ATR (основной, рекомендуемый)

SL и TP рассчитываются как кратные значению Average True Range (ATR-14).


| Параметр | Описание |
|---|---|
| ATR SL mult (1.5) | Stop Loss = цена ± ATR × 1.5 |
| ATR TP1 mult (1.5) | Take Profit 1 = цена ± ATR × 1.5 |
| ATR TP2 mult (3.0) | Take Profit 2 = цена ± ATR × 3.0  (при Use TP2 = ON) |
| Use TP2 | 50% позиции закрывается на TP1, 50% на TP2 |


## 3.2  Режим VWAP

Уровни строятся на основе стандартного отклонения цены относительно VWAP. Подходит для внутридневной торговли на 4H и 15M.

- SL = VWAP ± StdDev × 0.618 (по умолчанию)
- TP = VWAP ± StdDev × 1.0 (по умолчанию)


## 3.3  Режим ADX (адаптивный)

Stop Loss адаптируется под текущую фазу ADX.

- SLEEP: ATR × 0.75 — узкий стоп в боковике
- TREND: ATR × 1.5 — стандартный стоп в тренде
- OVER: ATR × 2.5 — широкий стоп при перегреве

# 4.  Система протоколирования (Log Export)

Система записывает два типа событий в Pine Editor → Logs. Данные экспортируются в CSV и анализируются в NotebookLM.


## 4.1  Торговые записи — закрытые сделки (log.info)

Запись создаётся при закрытии каждой сделки (по SL, TP1 или TP2). Включается через Enable Trade Log = ON.


Формат строки (19 полей):

```
date | L/S | entry | exit | profit% | bars_held | score |
vix  | vix_state | move | jpy% | macro | e2 | ipda% |
disc/prem | sweep | adx_phase | rvol | btc_state | WIN/LOSS
```


| Поле | Значение |
|---|---|
| date | Дата закрытия сделки (UTC, формат YYYY-MM-DD) |
| L/S | Направление: L = Long, S = Short |
| entry / exit | Цена входа и выхода |
| profit% | Прибыль/убыток в % от цены входа |
| bars_held | Количество баров в позиции |
| score | Confluence Score 0–6: количество выполненных условий |
| vix / vix_state | Значение VIX и статус: OK / WARN / PANIC |
| move | Значение индекса MOVE (волатильность облигаций) |
| jpy% | 5-дневное изменение USD/JPY в % |
| macro / e2 | 1 = фильтр выполнен, 0 = не выполнен |
| ipda% | Позиция цены в диапазоне: 0% = Low, 100% = High |
| disc/prem | Зона: DISC = Discount, PREM = Premium |
| sweep | 1 = weekly sweep сработал, 0 = нет |
| adx_phase | Фаза ADX: SLEEP / TREND / OVER |
| rvol | Относительный объём (1.0 = среднее) |
| btc_state | BTC_BULL = BTC выше SMA, BTC_BEAR = ниже |
| WIN / LOSS | Итог сделки по знаку profit% |


## 4.2  Диагностический аудит — заблокированные сигналы (log.warning)

Система фиксирует каждый потенциальный сигнал, заблокированный любым из фильтров. Это критически важно для понимания того, почему система молчала в конкретный момент.


| Тип записи | Условие блокировки | Что это означает |
|---|---|---|
| BLOCKED_SWEEP | Macro E1 OK, sweep = 0 | Недельный уровень не пробит — вход не подтверждён |
| BLOCKED_E2 | Sweep OK, VIX/MOVE/JPY заблокировал | Рынок в режиме страха или паники — вход опасен |
| BLOCKED_ADX | E2 OK, ADX фаза SLEEP или OVER | Нет тренда (консолидация) или тренд перегрет |
| BLOCKED_BTC | ADX OK, BTC медвежий или ETH слабее BTC | Крипторынок в неблагоприятном состоянии для ETH |

|---|---|---|---|---|
| VIX Filter ON (рекомендуется) | 188 | 54.26% | 1.53 | 3.31% |
| VIX Filter OFF (базовый) | 215 | 51.63% | 1.229 | 3.59% |

## 5.2  Workflow исследования через NotebookLM

Система интегрирована с NotebookLM через экспорт логов. Позволяет получать профессиональные инфографики и статистику без написания кода.


Пошаговый процесс сбора и анализа данных:

- Шаг 1: TradingView → Settings индикатора → Log Export → Enable Trade Log = ON
- Шаг 2: Дождаться пересчёта истории (происходит автоматически)
- Шаг 3: Pine Editor → вкладка Logs → выделить все строки → скопировать
- Шаг 4: Вставить в текстовый файл, сохранить с расширением .csv
- Шаг 5: Загрузить .csv в NotebookLM как новый источник
- Шаг 6: Задать вопросы: «Сколько WIN при VIX WARN?», «Средний RVOL у прибыльных сделок?»
- Шаг 7: После анализа — выключить Enable Trade Log = OFF
ВАЖНО: лог включать только для сбора данных. Постоянная запись снижает производительность индикатора.


## 5.3  Диагностический аудит блокировок

Помимо торговых записей, система пишет строки BLOCKED для каждого заблокированного сигнала. Это позволяет понять почему система молчала в конкретные периоды.


Вопросы для NotebookLM по диагностическому логу:

- «Какой фильтр заблокировал больше всего сигналов?» — найти наиболее ограничивающий фильтр
- «В какие месяцы было больше BLOCKED_E2?» — выявить сезонность волатильности
- «Были ли BLOCKED_SWEEP периоды когда цена потом выросла?» — оценить качество Sweep
- «Соотношение BLOCKED_ADX SLEEP vs OVER» — понять характер рыночных фаз
- «Сколько сигналов заблокировал BTC фильтр?» — оценить влияние крипто-корреляции


## 5.4  Управление таймфреймом

Стратегия корректно работает на любом ТФ благодаря адаптивному получению макроданных. При ТФ >= 1D данные VIX/MOVE/JPY берутся с текущего ТФ, при ТФ < 1D — принудительно с дневного.


| Таймфрейм | Данные VIX/MOVE | Рекомендация |
|---|---|---|
| 1W, 3D | Недельные / 3D | Стратегический уровень. Мало сигналов, высокое качество |
| 1D (основной) | Дневные | Оптимальный ТФ. Подтверждён статистикой 2023–2026 |
| 4H | Дневные (авто) | Рабочий ТФ. RVOL выключить, MOVE включить |
| 15M и ниже | Дневные (авто) | Много сигналов. Только для краткосрочного мониторинга |


## 5.5  Порядок активации фильтров

Никогда не включайте все фильтры одновременно. Активируйте последовательно, собирайте статистику, анализируйте в NotebookLM, принимайте решение на основе данных.


- Приоритет 1 — VIX = ON: уже статистически обоснован (38% win rate при WARN). Включить сразу.
- Приоритет 2 — MOVE: собрать 50+ сделок → загрузить лог → проанализировать влияние MOVE на результат
- Приоритет 3 — ADX: включить после анализа BLOCKED_ADX записей. Понять долю SLEEP и OVER
- Приоритет 4 — BTC: включить только после 100+ сделок с btc_state в логе
- Приоритет 5 — JPY: включать последним, только если JPY события видны в диагностическом логе
